To ensure conssitency in the face of keeper and backend failures, we implemented our system in a way where we use two backends to store the values for each key or list. These values are stored as a log and they are retreived by looking at the log and calculating the state of the value/list. In the case of the list log, we append a series of append or remove operations and use that to calculate the list's current state. We use the clock function as a method of figuring out which ops occured before others.

To ensure consistency across backends, we implemented the system such that each bin corresponds to one primary and one backup. If one of them goes down, the system defaults to bringing the next backend in the list of backends up to speed. Each keeper is keeping track of which backends are down and which are up. When a change is noticed, the keeper brings the backend up to speed if needed. To ensure that bad data is not sent, the latest data is always used by checking the recorded clock time from those operations. By taking the max clock time and data points always, we ensure that the system is always providing the most recent data.
